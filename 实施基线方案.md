# 实施基线方案：可迁移、本地化、可交付的校园 AI 安防平台

## 1. 当前新增约束
基于现有材料和你的补充，这个项目不是单校定制系统，而是要做成一套可复制交付的私有化产品。

### 核心约束
- 每个学校应支持独立部署。
- 允许按学校做少量配置或界面调整后复用。
- 数据尽量不出校，核心能力本地运行。
- 接入模型需要本地部署，不依赖外部云 API。
- 方案必须可实施、可运维、可升级、可审计。
- 最终成品要适合技术实施工程师交付，不是概念验证玩具。

## 2. 我对产品形态的判断
我当前的判断是：

- 不应做成一个“共享数据库的多租户 SaaS”。
- 应做成“统一产品基线 + 每校独立实例部署”的模式。
- 学校 A、学校 B 的差异，优先通过配置、字典、模型策略、设备映射解决，而不是改代码。

这个判断的原因很直接：

- 你强调本地化和隐私隔离。
- 校园安防属于高敏感场景。
- 后续学校之间的设备、点位、组织架构、告警策略一定不同。
- 私有化项目的交付稳定性，通常比共享平台更重要。

## 3. 不建议从 0 造大系统
这个项目可以自研业务层，但不应该从底层视频、媒体代理、识别服务、权限系统全部重写。

更稳的路线是：

- 选成熟开源项目做底座。
- 在其上做统一编排层和校园业务层。
- 把“可迁移”和“可交付”能力做成产品壳。

## 4. 推荐的系统拆分方式
建议把系统拆成 5 层：

1. 视频接入与转发层
   - 负责 RTSP / ONVIF / WebRTC / 录像与流转发。

2. 本地 AI 分析层
   - 负责人脸、目标检测、行为识别、情绪分析等模型推理。

3. 事件治理层
   - 负责把算法输出统一成事件、告警、处置流。

4. 平台业务层
   - 负责学校、校区、楼栋、摄像头、人员、策略、告警、报表、审计。

5. 运维交付层
   - 负责安装、升级、配置模板、学校迁移、日志与健康检查。

## 5. GitHub 基线建议
我的结论是：不建议押注一个单体 GitHub 项目，而是选 3 到 5 个成熟项目组合。

### 方案 A：推荐基线
这是当前最符合你目标的组合。

- 视频与边缘事件底座：Frigate
- 人脸识别服务：CompreFace
- 身份认证与权限：Keycloak
- 媒体代理与协议转发：MediaMTX
- 模型服务层：OpenVINO Model Server 或 Triton Inference Server
- 校园业务平台：自研二次开发层

### 为什么推荐这条路线
- 每一层边界清晰，后续替换成本低。
- 可以做到全本地部署。
- 大部分组件都有成熟容器化路径。
- 你后续做学校 B 时，主要改配置、组织结构、点位和规则。
- 业务平台能保持完全可控，不被某一个大项目绑死。

## 6. 各项目角色定位
### 6.1 Frigate
适合做本地视频分析/NVR 底座，不适合直接当最终校园业务平台。

它的价值在于：
- 本地运行。
- 已经具备视频流与 AI 事件处理基础。
- 适合作为摄像头接入和基础告警来源。

不建议把它当全部产品的原因：
- 它更偏视频安防底座，不是校园治理平台。
- 门禁、组织架构、心理研判、流程闭环、审计报表仍需要上层系统承接。

### 6.2 CompreFace
适合做人脸识别能力服务。

它的价值在于：
- 已经是可部署的人脸识别系统。
- REST API 友好，适合接进业务平台。
- 比从零做人脸服务更稳。

需要注意：
- 它是能力服务，不是门禁业务系统。
- 仍然需要你在上层做人员库、黑名单、通行策略、告警联动。

### 6.3 Keycloak
适合做统一认证、角色权限、学校级管理员分权、审计基础。

它的价值在于：
- 不必自己从零做登录、权限、令牌体系。
- 适合支撑学校管理员、平台管理员、心理老师、安保人员等角色体系。

需要注意：
- 不建议把“一个学校 = 一个 SaaS 租户”硬塞进复杂共享模型。
- 更建议每个学校实例独立部署，平台侧复用同一套角色模型。

### 6.4 MediaMTX
适合做流媒体路由层。

它的价值在于：
- 在多个分析模块、预览页面、录像模块之间做统一流转发。
- 后续接 WebRTC 实时预览时很有用。

### 6.5 OpenVINO Model Server / Triton
这两者更适合作为模型服务层，而不是直接做业务系统。

我的判断：
- 如果交付硬件以 Intel CPU/核显为主，优先 OpenVINO 路线。
- 如果交付硬件以 NVIDIA GPU 为主，优先 Triton 路线。

## 7. 推荐与不推荐的边界
### 推荐作为主路线
- Frigate：做视频/事件底座
- CompreFace：做人脸服务
- Keycloak：做认证授权
- MediaMTX：做流代理
- OpenVINO 或 Triton：做模型服务

### 只适合参考，不建议作为主基线
- Viseron
  - 本地化思路很好，但更适合自托管 NVR 场景，产品成熟度和企业交付感仍弱于组合式方案。

### 当前不建议作为主方案
- ZoneMinder
  - 历史长，但技术栈偏老。
  - GPL 许可对后续二次开发交付要谨慎评估。

## 8. 未来系统应该怎么做成“可迁移”
重点不是多租户，而是产品化配置能力。

建议从一开始就把下面这些做成配置而不是代码：

- 学校基础信息
- 校区/楼栋/楼层/点位结构
- 摄像头清单与映射
- 人员组织结构
- 黑白名单策略
- 告警分级规则
- 模型启停策略
- 本地存储策略
- 页面品牌与学校名称

这样学校 A 到学校 B 的迁移，才会是：

- 导入一套学校配置模板
- 接入本地摄像头和设备
- 导入组织架构和人员库
- 部署本地模型
- 调整告警策略

而不是重新改一轮代码。

## 9. 我建议的成熟落地架构
### 部署模式
- 每校一套独立部署实例。
- 统一容器化交付。
- 统一版本管理。
- 统一学校初始化脚本。
- 统一升级脚本。

### 交付形态
- `core-platform`：校园业务主平台
- `ai-gateway`：统一算法接入网关
- `device-gateway`：摄像头/门禁/电子卡等设备接入层
- `ops-installer`：部署、升级、巡检脚本

### 运维要求
- 所有服务支持离线部署。
- 所有配置支持导出导入。
- 所有关键事件支持审计留痕。
- 所有核心服务支持健康检查和重启恢复。

## 10. 技术路线判断
### 最合理的路线
不是寻找“一个 GitHub 仓库解决全部需求”，而是：

- 用成熟开源项目解决共性基础设施。
- 用你自己的业务层沉淀行业能力。
- 让最终可交付的部分掌握在自己手里。

### 为什么这是更稳的
- 单个仓库通常只能强在某一层。
- 你这个项目同时覆盖视频、算法、权限、安防流程、心理研判和实施交付。
- 真正值钱的是“平台编排层 + 可复制部署能力 + 校园业务模型”。

## 11. 当前最值得继续推进的方向
如果后面开始做第一版，我建议以这条路线继续：

1. 先做产品级技术选型确认。
2. 再确定推荐的开源组合和许可边界。
3. 再设计你的自研业务主平台边界。
4. 最后基于这个边界做首版 demo 和可部署工程骨架。

## 12. 需要你确认的几个关键假设
如果我后续继续推进，我将默认以下假设成立：

1. 第一版目标平台是 Web 管理后台，不是桌面端。
2. 第一版部署环境优先 Linux 服务器，不优先 Windows Server。
3. 每个学校独立部署一套实例，不做真正共享式多租户。
4. 第一版先支持本地摄像头和本地模型，不优先接复杂硬件联动。
5. 业务主平台允许自研，但基础能力优先复用开源项目。

如果这些假设有偏差，后面的技术路线要及时改。
