# Spec: 校园平台壳体 Demo 方案

## Assumptions
1. 当前目标是给甲方看“平台产品形态”，不是做完整可部署系统。
2. 当前阶段不接真实摄像头、不接真实模型、不接无人机。
3. 当前阶段重点是总控端、老师端、家长端三类角色的界面与权限分层。
4. 家长端现阶段优先做移动 H5 风格页面，不强制马上做微信小程序。
5. 第一版允许全部使用 Mock 数据，只要交互链路完整。

## Objective
快速做出一个看起来像成熟产品的校园平台 Demo，让甲方直观看到：

- 这是一个完整的平台，而不是几张静态页面。
- 总控端、老师端、家长端有清晰的角色差异。
- 权限、菜单、数据视图会随角色变化。
- 后续算法识别、设备接入、无人机联动都可以往这个平台壳体里挂。

## 推荐技术路线
### 已确认主线
用“统一设计体系 + 双端前端壳体”的方式实现。

- PC 端：
  - 总控端
  - 老师端
- Mobile 端：
  - 家长端

### 已确认 GitHub 基线
- PC 端基线：`ant-design/ant-design-pro`
- 移动端基线：`ant-design/ant-design-mobile`

确认原因：
- 视觉成熟，展示感更强。
- PC 管理端和移动端都有成熟组件体系。
- 更容易快速做出“像正式产品”的界面。
- 后续即使接真实后端和真实算法，前端壳体也不需要推翻。

### 备选路线
如果后面发现当前视觉方向不适合政企汇报，再考虑切换到 TDesign 体系：

- PC 端：`Tencent/tdesign-vue-next-starter`
- 移动端：`Tencent/tdesign-mobile-vue`

## 不推荐作为当前阶段主基线
### `marmelab/react-admin`
- 更偏数据后台框架。
- CRUD 很强，但“产品展示感”和多角色故事感不够强。

### `PanJiaChen/vue-element-admin`
- 经典，但视觉语言已经偏旧。
- 拿来快速改能跑，但不够像新项目的成熟方案展示。

### `vbenjs/vue-vben-admin`
- 工程能力强，适合后台产品。
- 但当前目标不是先做开发者友好的后台，而是先做甲方一眼能接受的整套产品外观。

## 系统架构建议
当前阶段不要先做后端大系统，先做“前端壳体 + Mock 服务层”。

### 推荐结构
- `apps/admin-web`
  - 总控端
  - 老师端
- `apps/parent-h5`
  - 家长端
- `packages/shared-ui`
  - 公共样式变量
  - 图标
  - 页面区块
- `packages/mock-server`
  - 登录
  - 用户信息
  - 菜单权限
  - 仪表盘数据
  - 消息通知

### 这样做的好处
- 演示快。
- 角色切换直观。
- 后续能逐步替换成真实接口。
- 不会因为后端未定而卡住整体产品展示。

## 角色设计
### 1. 总控端
定位：校方或平台管理层的全局视角。

核心页面：
- 平台总览
- 校园态势大屏
- 风险与告警中心
- 班级/年级/校区统计
- 用户与角色管理
- 系统配置
- 审计日志

展示重点：
- “全局可见”
- “多校区/多楼栋/多角色管理”
- “风险闭环”

### 2. 老师端
定位：日常班级管理与事件处理。

核心页面：
- 我的班级
- 学生动态
- 请假与到校情况
- 班级通知
- 待处理事件
- 家校沟通

展示重点：
- “面向班级日常工作”
- “信息聚焦，不是管理后台缩小版”
- “老师看到的是自己职责相关内容”

### 3. 家长端
定位：家长查看孩子信息与接收通知。

核心页面：
- 首页卡片
- 今日到校/离校状态
- 学校通知
- 请假申请
- 消息中心
- 联系老师
- 个人中心

展示重点：
- “轻量、可信、易懂”
- “家长最关心的是孩子状态和通知”
- “不是复杂后台，而是服务入口”

## 最适合甲方看的演示方式
不要一上来讲技术栈，先讲产品故事。

### 推荐演示链路
1. 登录页
   - 可切换“平台管理员 / 老师 / 家长”三种身份。
2. 登录为平台管理员
   - 展示平台总览和全局告警。
3. 切换到老师
   - 展示班级工作台和待处理事项。
4. 切换到家长
   - 展示孩子状态、通知、请假入口。

### 这条演示链路的价值
- 甲方会更容易理解“这不是单页原型，而是完整系统”。
- 能体现权限分层和产品成熟度。
- 后续技术能力可以自然挂接进来。

## 视觉风格建议
方向应是“校园政企感 + 稳定可信 + 不老气”。

- 主色建议：
  - 深蓝
  - 青蓝
  - 少量暖色告警色
- 总控端：
  - 偏深色或中性大屏风格都可
- 老师端：
  - 清晰、规整、信息密度中等
- 家长端：
  - 明亮、轻量、卡片式

注意：
- 三端风格应统一，但不能完全一样。
- 总控端是“管控感”。
- 老师端是“工作台感”。
- 家长端是“服务入口感”。

## Demo 边界
### 这一步要做
- 三端登录与角色切换
- 三端首页
- 基础菜单与权限差异
- 典型业务页面骨架
- Mock 数据联动

### 这一步不要做
- 真实设备接入
- 真实 AI 推理
- 真正的复杂后端权限中心
- 真正的多学校部署
- 真正的数据库设计定稿

## Commands
如果后续按当前路线实现，我建议统一用 `pnpm` 和 monorepo。

- 安装：`pnpm install`
- 管理端开发：`pnpm --filter admin-web dev`
- 家长端开发：`pnpm --filter parent-h5 dev`
- 全量启动：`pnpm dev`
- 构建：`pnpm build`

## Code Style
```ts
export const roleHomeMap = {
  admin: '/overview',
  teacher: '/classroom',
  parent: '/child-home',
} as const;

export function resolveHomeByRole(role: keyof typeof roleHomeMap) {
  return roleHomeMap[role];
}
```

约束：
- 路由、菜单、权限用配置驱动。
- 页面壳优先，复杂逻辑延后。
- 所有接口先抽象成服务层，再决定是否接真实后端。

## Testing Strategy
当前阶段以“可演示验证”为主，不以完整自动化测试为先。

- 页面级验证：
  - 角色切换后菜单变化正确
  - 三端首页能独立展示
  - Mock 数据可以正常渲染
- 交互级验证：
  - 登录
  - 页面跳转
  - 通知、待办、告警的基础联动

## Boundaries
- Always:
  - 先保证产品壳完整
  - 所有页面都可导航
  - 角色差异必须明确
- Ask first:
  - 是否要一步到位做小程序
  - 是否要先接真实接口
  - 是否要用现成大屏模板
- Never:
  - 在这一阶段陷入算法实现
  - 在这一阶段深挖部署与硬件联调
  - 为了“看起来真”而把架构做重

## Success Criteria
- 甲方能在 3 到 5 分钟内看懂三类角色的区别。
- Demo 看起来像一个完整平台，而不是几张独立页面。
- 后续博士团队做视觉能力时，可以自然挂接到总控端和老师端页面里。
- 你后续如果决定继续开发，这套前端壳体不需要推翻重来。

## Open Questions
1. 家长端后续更偏 H5 还是微信小程序？
2. 甲方更偏喜欢“深色大屏”还是“明亮管理后台”？
3. 第一版是否需要体现“多学校切换”，还是先只展示单校？

## Decision Record
- 2026-06-19：前端壳体基线确认采用 `ant-design/ant-design-pro` 和 `ant-design/ant-design-mobile`。
- 原因：当前阶段以视觉展示、角色分层和产品完整感优先，优先选择更容易快速出效果的方案。
