# 当前代码架构索引

## 1. 目标
这份索引不是重新写方案，而是把当前仓库里已经落地的代码结构固定下来，作为后续继续补功能的实现基线。

当前目标是：
- 让管理端、老师端、家长端都有稳定目录边界
- 让 Mock 数据层尽量接近未来真实接口替换方式
- 让后续功能新增以“填模块”为主，而不是继续拆地基

## 3. admin-web 边界

### 3.1 `modules/*`
这里放真实页面实现。

当前已经稳定到：
- `overview`
- `access-control`
- `patrol`
- `emotion`
- `alert-center`
- `system-config`
- `teacher-workbench`

### 3.2 `pages/*`
这里只做路由适配，不再承载业务实现。

作用：
- 兼容 Umi 现有路由扫描方式
- 后续即使模块目录继续细化，也不影响已有路由路径

### 3.3 `demo/*`
作为本轮 Mock 数据接入层。

职责：
- `entities`：基础实体和事件类型
- `events`：统一事件流
- `views/index.ts`：基础聚合视图与兼容导出
- `views/admin.ts`：总控侧/配置侧聚合视图
- `views/teacher.ts`：老师侧聚合视图

后续接真实接口时，优先替换这里，不动页面组件。

## 4. parent-h5 边界

### 4.1 `app/AppShell.tsx`
负责：
- tab 导航状态
- 弹层开关
- 顶层页面切换

不负责：
- 每个 tab 的具体业务内容

### 4.2 `modules/*`
当前按家长端核心语义拆分：
- `home`
- `child-status`
- `messages`
- `services`
- `leave`
- `profile`

这样后续补“请假记录”“反馈建议”“消息详情”时，可以直接在对应模块内扩展，不需要继续改 `App.tsx`。

### 4.3 `demo/*`
当前是家长端本地演示数据层。

建议后续继续保持：
- 页面只拿聚合后的 view
- 页面不要直接散落写临时假数据

## 5. 当前已稳定的核心原则

1. 管理端与老师端共用 `admin-web`，靠角色分流和模块划分承载。
2. 家长端单独保留 `parent-h5`，贴近移动端演示形态。
3. 三端仍围绕统一事件故事线表达，不进入真实模型/设备集成。
4. 当前阶段以 Mock-first 为主，但 Mock 结构已按未来可替换边界组织。

## 6. 后续推荐新增顺序

### admin-web
1. 告警中心详情页
2. 门禁二级页：异常告警 / 点位详情
3. 巡检二级页：重点区域 / 任务状态
4. 老师端：学生详情 / 通知详情 / 待办详情

### parent-h5
1. 请假记录
2. 消息详情
3. 反馈建议
4. 服务入口的更多业务卡片

## 7. 不建议再做的事

1. 不要再把大量业务实现写回 `pages/*`
2. 不要再把家长端新功能堆回单个 `App.tsx`
3. 不要在页面内部直接复制新的假数据对象
4. 不要在本轮引入真实后端耦合，破坏当前 Demo 可控性

## 8. 结论
当前仓库已经从“原型页面集合”进入“可持续扩展的 Demo 骨架”阶段。

后续如果继续推进，建议始终遵守一句话：

先补模块，再补页面；先补视图聚合，再补接口接入。
