# Spec: 校园协同平台 Demo 第二阶段升级

## Assumptions
1. 当前目标仍然是“高完成度 Demo”，不是接真实设备、真实模型或真实后端。
2. 当前升级重点是补齐一期业务表达，不是扩到二期电子卡、无人机、公安互通。
3. 本轮继续以 `admin-web` 和 `parent-h5` 为主，不新增桌面端或独立老师 App。
4. 老师端仍然挂在 `admin-web` 内，以角色分流方式承载。
5. 所有 AI、门禁、巡检、情绪识别能力本轮都使用统一 Mock 事件流表达。
6. 本轮允许继续优化目录结构，但不引入会改变本机全局环境的新工具链。

如果这些假设有偏差，后续实现顺序和范围要及时调整。

## Objective
基于现有三端壳体，升级成一套更接近“一期校园平台”的 Demo。目标不是只展示角色外观，而是把以下业务能力补齐到可讲、可演、可联动的程度：

- 总控端能展示校园态势、门禁通行、巡检告警、情绪研判、统一告警中心。
- 老师端能承接待办、班级动态、心理复核、家校通知等日常闭环。
- 家长端能展示孩子状态、通知消息、请假申请、服务反馈。
- 三端围绕同一套事件和角色数据形成完整故事线。

## Current State
当前已经完成：
- 管理端登录页、角色切换、总控端基本首页、风险中心、系统配置。
- 老师端工作台、我的班级、待办中心。
- 家长端首页、孩子、消息、服务、请假申请。
- 三端共享基础故事线和演示脚本。

当前还缺：
- 与“项目方向文档”更直接对应的一期模块命名和页面表达。
- 门禁通行、人脸/陌生人、校园巡检、群体情绪等核心业务页。
- 统一告警中心和事件流转状态页。
- 更清晰的角色间联动关系和页面跳转关系。
- 更完整的 Mock 数据模型与事件生命周期。

## Upgrade Scope
### 1. 总控端补齐
- 校园态势总览大屏
- 门禁管理 / 通行记录
- 校园巡检 / 风险告警
- 情绪研判 / 群体趋势
- 统一告警中心

### 2. 老师端补齐
- 学生动态页
- 班级通知页
- 心理关注复核页
- 待办处理详情页

### 3. 家长端补齐
- 通知详情页
- 请假记录页
- 反馈建议页
- 与老师端对应的消息链路

### 4. 数据与联动补齐
- 定义统一事件类型
- 定义统一角色和组织结构
- 定义事件状态流转
- 定义页面间跳转和来源关系

## Testing Strategy
本轮以“业务表达完整 + 交互链路稳定”为主：

- 管理端：
  - `npm.cmd run tsc`
  - `npm.cmd run test`
  - 关键页面手动走查
- 家长端：
  - `npm.cmd run build`
  - 关键表单与弹层手动走查
- 联动验证：
  - 同一事件能在总控端、老师端、家长端找到对应表现
  - 页面跳转关系正确
  - 演示脚本可顺利执行

## Boundaries
- Always:
  - 优先对齐“项目方向文档”中的一期重点
  - 所有新增功能先以 Mock 演示方式实现
  - 三端联动必须清晰
- Ask first:
  - 是否要把老师端从 `admin-web` 独立成单独前端
  - 是否要新增 workspace / packages 结构
  - 是否要开始接真实接口
- Never:
  - 在本轮引入真实视觉识别实现
  - 在本轮进入硬件联调
  - 在本轮把 Demo 做成重后端项目

## Success Criteria
- 总控端能覆盖“门禁、巡检、情绪、告警”四类一期能力表达。
- 老师端能承接至少一条完整闭环：家长请假 -> 老师待办 -> 总控可见。
- 家长端能覆盖状态查看、通知接收、服务提交三类核心入口。
- 三端共享统一事件模型，甲方能清楚理解系统不是三套孤立页面。
- 后续接真实模型或真实设备时，不需要推翻当前前端结构。
